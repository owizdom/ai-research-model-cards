# Charts — benchmark overlap analyses

Two follow-on analyses that extend the homepage fragmentation finding with a
recency cut and a temporal trajectory.

## What's here

| File | What it is |
|---|---|
| `overlap_2026_heatmap.svg` / `.png` | Lab × Lab benchmark Jaccard for cards released in 2026 |
| `overlap_over_time.svg` / `.png` | % of distinct benchmarks reported by ≥2 labs, trailing 4-quarter window |
| `data/overlap_2026.csv` | Underlying pairwise overlap (lab_a, lab_b, n_intersect, n_union, jaccard) |
| `data/overlap_over_time.csv` | Per-quarter rollup (n_cards, n_labs, n_distinct_benchmarks, n_shared_2plus, pct_shared) |
| `data/snapshot_2026-04-28.json` | Frozen pull from the live Railway API used as input |
| `release_dates.csv` | Hand-curated publication date per `document_slug` (the analytical bedrock) |
| `generate.py` | Builds everything above from snapshot + dates |

## Headline findings

### % shared has been declining for two years

From the trailing 4-quarter rollup:

| Quarter | % shared | Cards in window |
|---|---:|---:|
| 2024-Q1 | **24%** | 6 |
| 2024-Q4 | 20% | 15 |
| 2025-Q2 | 12% | 20 |
| 2025-Q4 | 9% | 24 |
| 2026-Q2 | **6%** | 19 |

As more cards have been published, the share of benchmarks that ≥2 labs both
report has dropped from ~24% to ~6%. Lab-equal weighting (each lab contributes
one benchmark set per window regardless of how many cards it shipped) and
benchmark family collapse (mmlu_pro→mmlu, gpqa_diamond→gpqa, …) hold across
the entire series, so the trend is not a naming artifact.

### 2026 cards: zero pairwise overlap

The 6 model cards released in 2026 so far (3 Anthropic, 2 OpenAI, 1 Google)
share **no benchmarks** at the family level across any pair of labs. This is
real, but it is partly a *document-mix* effect, not pure divergence:

- **Anthropic (3 cards, 43 benchmarks)** ships comprehensive system cards mixing
  capability (MMLU, GPQA, MATH, SWE-bench, AIME, HLE, MMMU, OSWorld) with
  internal safety + agentic evals.
- **Google (1 card, 7 benchmarks)** — the Gemini 3.1 Pro card is a *safety-only*
  document with FSF evaluations (`fsf_harmful_manipulation_evaluation`,
  `multilingual_safety`, `text_to_text_safety`, …). Capability benchmarks for
  this generation presumably live in a separate technical report not yet
  collected.
- **OpenAI (2 cards, 10 benchmarks)** — GPT-5.3 Codex card is coding-focused
  (`cyber_safety_eval_*`, `network_attack_simulation`, `sabotage_suite`); GPT-5.5
  has only one scored eval (`apollo_sabotage_tasks`).

Read alongside the over-time chart, this suggests labs are drifting toward
publishing narrower, more lab-specific cards — when they overlap on a
benchmark at all, they're trending toward not doing so.

## Method

### Source

Data is pulled from the live Railway API at
`https://modest-playfulness-production.up.railway.app/api/v1/`. The frozen
snapshot in `data/snapshot_*.json` is the input of record — committing it
guarantees the chart can be regenerated byte-for-byte even if the API drifts.

The live API returns 6 labs / 50 model cards / 1,056 evals. METHODOLOGY.md
references 9 labs / 88 docs / 1,417 evals — that figure includes additional
labs (Cohere, Amazon, AI21) and document types (`constitution`,
`usage_policy`) that are tracked but either not yet exposed via this API
endpoint or filtered to model_card here. Both numbers are correct for their
respective scope.

### Filters

- `doc_type == 'model_card'` (excludes `constitution`, `usage_policy`).
- `state == 'scored'` (a row needs an explicit numeric score). Protocol-v1
  rows had no `state` field; they are treated as `scored` since v1 only
  emitted scored rows.
- Benchmark slugs are family-collapsed via the same rules used by
  `apps/api/src/api/v1/evals.py:117` (`FAMILY_SQL_EXPR`) so the chart is
  comparable to the homepage fragmentation finding.

### Metric definitions

- **% shared** = 1 − (n benchmarks reported by exactly one lab) / (n distinct
  benchmarks reported in window). Direct inverse of the existing `pct_unique`.
- **Pairwise Jaccard** (heatmap only) = |B_i ∩ B_j| / |B_i ∪ B_j|, where
  B_lab is the union of family-collapsed benchmark slugs across all of that
  lab's cards in the window.
- **Lab-equal weighting** — each lab contributes one set, regardless of card
  count. Anthropic's 6 cards do not get 6× the voting weight over Mistral's 1.
- **Trailing 4-quarter window** — at quarter Q, include all cards released in
  [Q-3, Q]. Smooths over lumpy publication cadence.

### Time anchor

`document_versions.version_date` from the live DB is uniformly clustered in
2026-03 / 2026-04 (bulk-import dates from a recent rebuild) and is unusable as
a publication-time anchor. `model_generations.release_date` is mostly NULL.
**`release_dates.csv` is the source of record** — 50 rows, with a `confidence`
column distinguishing high (URL-encoded or verifiable public announcement) /
med (estimated to month) / low (post training-cutoff, estimated to quarter).

Of the 6 cards released in 2026, 5 have `low` confidence (post Jan 2026
training cutoff for the model that built this CSV). If you have ground-truth
release dates for any of these, edit `release_dates.csv` and rerun
`python3 charts/generate.py` — the chart updates with no other code changes.

## How to regenerate

From the repo root:

```bash
# Replay (deterministic, network-free, uses latest committed snapshot)
python3 charts/generate.py

# Refresh (re-pull from live API, write a new snapshot, regenerate)
python3 charts/generate.py --refresh
```

`generate.py` has no third-party imports outside `matplotlib`. Install with:

```bash
python3 -m pip install --user matplotlib
```

## Caveats worth surfacing

1. **Small N for 2026.** 6 cards, 3 labs. The pairwise heatmap is an
   illustration of state-of-the-world-as-of-now, not a stable estimate.
   The trailing-window line chart absorbs this noise more gracefully.
2. **Document-mix selection effect.** Different labs publish different
   *kinds* of cards in any given window. Overlap is partly a function of
   what each lab chose to release as a "model card" that quarter. Anthropic
   bundles capability + safety; Google increasingly publishes safety-only
   cards; OpenAI has split into specialized cards (Codex, Operator, etc.).
3. **30 KB extraction window.** Per `claims/system_audit.md`, recall on
   long Anthropic system cards is incomplete (~48-78%). Some of the
   "anthropic-only" benchmarks may exist in other labs' cards but past
   the extraction window. The chart should be read as a lower bound on
   shared reporting.
4. **Family collapse trades nuance for comparability.** mmlu_pro and mmlu
   measure different things in practice. The chart treats them as the
   same family because the homepage analysis does. A "raw slug" version
   would show even less overlap.
5. **Hand-curated dates.** 5 of the 6 2026 cards have low-confidence dates
   (post training cutoff). The quarterly bin is robust to ±2 month error;
   anything finer is not.
